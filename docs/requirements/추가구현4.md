# 추가 구현 4 - 아키텍처 개선 (Design Patterns)

본 문서는 메신저 시스템의 코드 유지보수성과 확장성을 높이기 위해 적용 가능한 디자인 패턴 활용 방안을 정의합니다.

## 공통 사항
* 본 문서의 내용은 특정 API 스펙보다는 시스템 내부의 구조적 개선에 중점을 둡니다.

---

## 아키텍처 개선 (Design Patterns)

> 코드의 유지보수성과 확장성을 높이기 위해 적용된 디자인 패턴입니다.

### Observer 패턴 활용 (UI와 로직의 분리)

*   **목적**: 클라이언트 애플리케이션에서 메시지 수신 로직과 UI 업데이트 로직을 분리하여 상호 의존성을 낮춥니다.
*   **적용 방안**:
    *   `MessageReceiver`(Subject)는 서버로부터 새로운 메시지를 수신하면 등록된 `MessageObserver`(Observer)들에게 알림을 보냅니다.
    *   UI 컴포넌트(예: 채팅창, 알림 바)는 Observer 인터페이스를 구현하여 수신된 메시지를 화면에 출력하거나 알림을 띄우는 등 각자의 역할을 수행합니다.
    *   이를 통해 새로운 UI 기능이 추가되어도 핵심 메시지 수신 로직을 수정할 필요가 없습니다.

### Command 패턴 적용 (채팅 명령 처리 분리)

*   **목적**: 사용자가 입력하는 다양한 채팅 명령어(예: `/list`, `/exit`, `/w`)를 객체화하여 처리 로직을 캡슐화하고 확장성을 확보합니다.
*   **적용 방안**:
    *   `ChatCommand` 인터페이스를 정의하고, 각 명령어별로(예: `ListCommand`, `WhisperCommand`) 이를 구현합니다.
    *   사용자 입력이 들어오면 `CommandFactory`가 명령어를 파싱하여 적절한 Command 객체를 생성합니다.
    *   `CommandExecutor`는 생성된 Command 객체의 `execute()` 메서드를 호출하여 로직을 실행합니다.
    *   새로운 명령어가 추가되더라도 기존의 조건문(if-else)을 수정하지 않고 새로운 클래스를 추가하는 것만으로 기능을 확장할 수 있습니다.
